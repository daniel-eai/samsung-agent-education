현재 프로젝트의 CHANGE_REQUEST.md, AORR.md, MEMORY.md,
현재 코드, 사용자 참고 자료와 마지막 정상 배포 commit을 읽고
Step 8에서 설계한 사용자 변경 요청에 대한 Self-Correcting TDD 재루프를 실행해줘.

이번 단계의 목표는 CHANGE_REQUEST.md에 정의된 모든 Change Item을
의존성 순서대로 구현하고 검증하는 것이야.

특정 게임 기능이나 특정 웹사이트 변경에 한정하지 마.

사용자의 요청이 버그 수정, UI/UX 개선, 콘텐츠 변경, CV 기반 콘텐츠 작성,
페이지 구조 변경, 멀티페이지 전환, 신규 게임 기능, 성능 개선,
접근성 개선 등 어떤 형태이든 Step 8에서 설계된 변경 루프에 따라 처리해.

기존 정상 기능, 완료 기록, 실행 기록과 마지막 정상 배포 정보는
삭제하거나 덮어쓰지 마.

## 1. 변경 실행 전 확인

먼저 다음을 확인해:

- 전체 Change Request ID
- 모든 Change Item과 현재 상태
- 사용자 요청 원문
- 참고 자료
- 현재 동작과 기대 동작
- Change Item별 완료 기준
- 변경 허용 범위
- 변경 금지 범위
- Change Item 간 의존성
- 실행 순서
- 마지막 정상 배포 commit
- 마지막 정상 배포 URL
- 현재 commit
- 현재 Git 상태
- 기존 테스트와 회귀 테스트
- 현재 Retry 기록
- Rollback 기준

CHANGE_REQUEST.md와 AORR.md의 실행 순서가 일치하지 않으면
임의로 진행하지 말고 HITL_REQUIRED로 중지해.

## 2. 기준선과 복구 지점 확보

코드를 수정하기 전에 다음을 MEMORY.md에 기록해:

- 변경 전 commit hash
- 마지막 정상 배포 commit
- 마지막 정상 배포 URL
- 현재 Git 상태
- 기존 테스트 결과
- 수정 전 웹사이트 상태
- 수정 전 게임 상태
- Rollback 기준

사용자 승인 없이 다음을 수행하지 마:

- hard reset
- force push
- 기존 commit 삭제
- 기존 정상 배포 덮어쓰기
- 브랜치 기록 재작성

## 3. Change Item별 루프 실행

CHANGE_REQUEST.md에 정의된 Change Item을 의존성 순서대로 실행해.

각 Change Item마다 독립적인 AORR 루프를 수행해:

### Act

- 수정 전 문제 또는 현재 동작 재현
- 완료 기준에 필요한 최소 변경 수행
- 승인된 범위의 파일만 수정
- 필요한 경우 테스트 또는 검증 스크립트 추가
- 관련 문서가 있다면 문서에서 확인된 사실만 반영

### Observe

- 수정 전과 수정 후 결과 비교
- 테스트 결과
- exit code
- 브라우저 콘솔
- 네트워크 및 파일 로드
- 반응형 화면
- 입력 반응
- 게임 상태
- 페이지 이동
- 콘텐츠 정확성
- GitHub Pages 호환성

### Reason

실패 원인을 다음 중 하나 이상으로 분류해:

- HTML_STRUCTURE
- CSS_RESPONSIVE
- JAVASCRIPT
- PERFORMANCE
- INPUT_LATENCY
- GAME_STATE
- GAME_LOGIC
- GAME_RENDERING
- GAME_CONTROL
- GAME_ENTITY
- GAME_EFFECT
- CONTENT
- DOCUMENT_EXTRACTION
- INFORMATION_ARCHITECTURE
- MULTI_PAGE_STRUCTURE
- NAVIGATION
- ACCESSIBILITY
- TEST
- ENVIRONMENT
- GITHUB_PERMISSION
- DEPLOYMENT
- UNKNOWN

### Repeat

- 한 Retry에서는 하나의 실패 원인만 수정
- 수정 후 동일한 Verifier 재실행
- 기존에 통과한 관련 기능 회귀 테스트
- MEMORY.md에 가설, 변경 파일, 결과와 오류 fingerprint 기록

## 4. 원자적 변경 원칙

기본적으로 각 루프에서는 최소 변경을 수행해.

다만 다음과 같이 여러 파일이 함께 변경되어야만 정상 동작하는 경우에는
하나의 원자적 변경 루프로 처리해:

- 단일 페이지를 멀티페이지로 변경
- 공통 내비게이션과 여러 페이지 생성
- 콘텐츠를 여러 페이지로 이동
- HTML, CSS, JavaScript가 함께 변경되어야 하는 신규 기능
- 게임 상태, 렌더링, 입력 처리가 함께 필요한 신규 기능
- 공통 컴포넌트 또는 공통 스타일 변경

원자적 변경 중 일부 파일만 수정된 불완전한 상태를
PASSED로 처리하지 마.

## 5. 참고 문서 기반 구현

사용자가 CV, 이력서, PDF 또는 다른 문서를 기반으로 콘텐츠를 변경하라고 요청한 경우:

1. 실제 파일을 읽고 사실 추출
2. 출처가 확인되는 정보만 사용
3. 사용자의 이름, 경력, 학력, 연구, 프로젝트, 논문, 기술 등을 적절한 페이지에 배치
4. 중복 정보 정리
5. 문서에 없는 내용 생성 금지
6. 공개 여부가 불분명한 개인정보는 [사람 확인 필요]
7. 콘텐츠를 축약하거나 표현을 수정한 경우 원래 의미를 유지
8. 페이지 제목과 사이트 타이틀도 문서 내용에 근거해 작성

파일이 없거나 손상되었거나 내용을 확인할 수 없으면
DOCUMENT_EXTRACTION 또는 HITL_REQUIRED로 처리해.

## 6. 구조 변경 구현

멀티페이지, 메뉴 변경 또는 정보 구조 변경이 포함된 경우 다음을 검증해:

- 필요한 HTML 페이지 생성
- 각 페이지의 title과 meta viewport
- 공통 내비게이션
- 현재 페이지 표시
- 페이지 간 상대 링크
- 홈으로 돌아가는 경로
- 공통 CSS 및 JavaScript 연결
- 모바일 메뉴
- 기존 콘텐츠 이동 완료
- 콘텐츠 중복 또는 누락 없음
- 깨진 링크 없음
- GitHub Pages 경로 호환성
- 직접 URL로 페이지 접근 가능
- 브라우저 뒤로 가기와 앞으로 가기 동작

## 7. 성능 및 입력 문제 검증

사용자 요청에 반응 속도, 입력 지연 또는 성능 문제가 포함된 경우 다음을 확인해:

- 이벤트 리스너 중복 여부
- 게임 루프 중복 여부
- 입력 큐 또는 방향 전환 처리
- touchstart, pointerdown, click 이벤트 선택
- preventDefault 적용 범위
- 불필요한 지연이나 debounce
- 메인 스레드 블로킹
- 렌더링 주기
- 입력 후 상태 반영 시점

“더 빠르게”와 같은 요청은 가능한 경우
수정 전후의 객관적인 관찰 결과를 기록해.

## 8. 신규 기능 검증

새로운 기능이 포함되어 있다면 다음을 정의하고 검증해:

- 생성 조건
- 초기 상태
- 동작 규칙
- 사용자와의 상호작용
- 점수 또는 상태에 미치는 영향
- 종료 또는 제거 조건
- 다시 시작할 때의 초기화
- 일시정지 시 동작
- 모바일 및 데스크톱 입력
- 기존 기능과의 충돌
- 중복 생성 방지
- 성능 영향
- 접근성 또는 설명 UI

사용자 요청에 명시되지 않은 동작 규칙이 필요하면
임의로 중요한 게임 규칙을 확정하지 말고 [사람 확인 필요]로 표시해.

## 9. 시각 효과 검증

애니메이션, 흔들림, 반전, 파티클, 아이템 효과 등 시각 효과가 포함된 경우:

- 효과가 시작되는 조건
- 효과의 지속 시간
- 반복 여부
- 게임 상태에 미치는 영향
- 효과 종료 후 원래 상태 복구
- 모바일 성능
- prefers-reduced-motion 처리
- 색상 대비
- 효과 중 입력 가능 여부
- 동일 효과의 중복 실행 방지

시각 효과가 기능 로직을 방해하면 PASSED로 처리하지 마.

## 10. Retry 정책

각 오류에 대해 다음 정책을 적용해:

- 하나의 오류당 최대 3회 Retry
- 동일 오류 fingerprint가 2회 반복되면 즉시 중지
- 한 Retry에서 하나의 실패 원인만 수정
- 테스트 삭제 금지
- 완료 기준 완화 금지
- 기존 기능 삭제를 통한 우회 금지
- 사용자 요청 누락 금지
- 문서에 없는 사실 생성 금지
- 불필요한 전체 프로젝트 재작성 금지
- 환경이나 권한 문제를 코드로 우회하지 않음

Retry 한계에 도달한 Change Item은 BLOCKED 또는 HITL_REQUIRED로 표시해.

하나의 Change Item이 중단되어도 독립적인 다른 Change Item을
안전하게 실행할 수 있다면 계속 진행할 수 있어.

단, 선행 의존성이 실패한 후속 Change Item은 실행하지 말고 BLOCKED로 표시해.

## 11. 범위 확장 처리

구현 도중 CHANGE_REQUEST.md에 없던 추가 변경이 필요하다고 판단되면:

- 필요한 이유
- 추가 파일
- 추가 위험
- 기존 완료 기준에 미치는 영향
- 추가 테스트

를 기록해.

사소하고 필수적인 연쇄 수정은 같은 루프에 포함할 수 있어.

그러나 다음에 해당하면 임의로 범위를 확장하지 말고 HITL_REQUIRED로 중지해:

- 새로운 프레임워크 도입
- 대규모 구조 변경
- 사용자 요청에 없는 콘텐츠 삭제
- 외부 API 또는 서비스 추가
- URL 체계의 추가 변경
- 개인정보 공개
- 기존 기능 제거
- 배포 방식 변경

## 12. Change Item별 완료 처리

각 Change Item은 다음 중 하나의 상태를 가져야 해:

- READY
- ACTING
- VERIFYING
- RETRYING
- PASSED
- BLOCKED
- HITL_REQUIRED

다음 조건을 모두 만족해야 PASSED로 표시해:

- 사용자 요청 반영
- 완료 기준 충족
- 변경 후 테스트 통과
- 관련 회귀 테스트 통과
- 문서 및 실행 기록 업데이트
- 다른 Change Item을 깨뜨리지 않음

부분적으로 구현된 Change Item을 PASSED로 표시하지 마.

## 13. 전체 회귀 테스트

모든 실행 가능한 Change Item이 완료된 후 전체 회귀 테스트를 수행해.

### 웹사이트

- 모든 페이지 로드
- Home 또는 메인 페이지
- About
- Experience
- Projects 또는 Research
- Games
- Contact
- 사용자가 요청한 추가 페이지
- 공통 내비게이션
- 모바일 내비게이션
- 현재 페이지 표시
- 외부 링크
- 내부 링크
- 콘텐츠 누락
- 브라우저 콘솔 오류
- 정적 파일 로드

실제 페이지 이름은 현재 프로젝트와 사용자 요청을 기준으로 판단하고,
위 이름을 하드코딩하지 마.

### 반응형

가능하면 다음 viewport에서 검증해:

- 태블릿: 약 768px

확인 항목:

- 가로 스크롤 없음
- 텍스트 잘림 없음
- 내비게이션 정상
- 버튼과 링크 터치 가능

- 게임 화면과 조작 UI 정상

### 게임

프로젝트에 실제 구현된 모든 기존 및 신규 게임 기능을 검증해:

- 게임 시작
- 게임 오버 후 재시작
- 마우스 또는 터치 시작
- 키보드 조작
- 모바일 조작
- 사용자 요청에 정의된 키 조작
- 일시정지와 재개
- 점수
- 최고 점수
- 음식 또는 아이템
- 충돌 처리
- 신규 엔티티
- 신규 시각 효과
- 색상 또는 테마 상태
- 중복 타이머 방지
- 입력 반응
- 모바일 배치
- Games 페이지 재진입

사용자 요청에 없는 특정 기능을 필수라고 가정하지 말고

### 접근성

## 14. Claude Code CLI

가능하면 Claude Code CLI를 독립 Verifier로 사용해.

Claude Code CLI에서 Sonnet 5가 실제로 사용 가능한지 먼저 확인해.
사용 가능하면 Sonnet 5를 사용해.

사용할 수 없다면 현재 사용 가능한 Sonnet 모델을 사용하고
실제 사용 모델명을 MEMORY.md에 기록해.

모델명을 임의로 추측하지 마.

Claude Code CLI를 사용할 수 없다면 사유를 기록하고
사용 가능한 다른 Verifier로 계속 진행해.

## 15. 기록 업데이트

각 루프가 끝날 때마다 다음을 업데이트해:

### CHANGE_REQUEST.md

- Change Item 상태
- 실제 수정 파일
- 테스트 결과
- Retry 횟수
- 완료 또는 중지 이유
- 사람 확인 필요 항목

### AORR.md

- 실행한 Change Loop
- 상태 전이
- 실패 원인
- Retry 결과
- Stop 또는 HITL 사유

### MEMORY.md

- Loop ID
- Change Item ID
- 시작 및 종료 상태
- 가설
- Act
- 변경 파일
- Verifier
- 결과
- exit code
- 오류 fingerprint
- Retry 횟수
- 다음 Loop
- 현재 정상 commit 후보
- Rollback 기준

기존 기록은 삭제하거나 덮어쓰지 마.

## 16. 로컬 변경 완료 조건

다음 조건을 모두 만족한 경우에만 전체 변경 구현이 완료된 것으로 판단해:

- 모든 실행 대상 Change Item이 PASSED
- HITL_REQUIRED 항목이 남아 있지 않음
- 모든 완료 기준 충족
- 전체 웹사이트 회귀 테스트 통과
- 전체 게임 회귀 테스트 통과
- 모바일, 태블릿, 데스크톱 검증 완료
- 콘텐츠 정확성 검증 완료
- 브라우저 콘솔에 치명적인 오류 없음
- 깨진 링크 없음
- GitHub Pages 호환성 검증 완료
- CHANGE_REQUEST.md 업데이트 완료
- AORR.md 업데이트 완료
- MEMORY.md 업데이트 완료

하나라도 충족되지 않으면
DEPLOY_APPROVAL_REQUIRED 또는 DEPLOYED로 표시하지 마.

모든 로컬 변경과 검증이 완료되면 상태를
DEPLOY_APPROVAL_REQUIRED로 설정해.

## 17. 사용자 재배포 승인

모든 로컬 변경이 완료되면 아직 commit, push 또는 배포하지 마.

Git remote를 확인해 실제 저장소와 배포 주소를 확인한 후
다음 형식으로 사용자에게 질문하고 중지해:

“사용자 수정 요청에 대한 모든 실행 가능한 변경 루프와 로컬 검증이 완료되었습니다.

Change Request:
[전체 Change Request ID]

완료한 항목:
[Change Item별 완료 내용]

변경 파일:
[변경 파일 목록]

보류 또는 사람 확인 필요 항목:
[없음 또는 항목 목록]

대상 저장소:
https://github.com/[GitHub 사용자명]/[GitHub 사용자명].github.io

예상 배포 주소:
https://[GitHub 사용자명].github.io

현재 수정 버전을 GitHub Pages에 재배포할까요?”

현재 저장소가
https://github.com/daniel-eai/daniel-eai.github.io.git 이라면 다음 주소를 사용해:

- 저장소: https://github.com/daniel-eai/daniel-eai.github.io
- 배포 주소: https://daniel-eai.github.io

사용자가 명시적으로 승인하기 전에는 다음을 하지 마:

- commit
- push
- 재배포
- GitHub Pages 설정 변경
- Git remote 변경

## 18. 승인 후 재배포

사용자가 명시적으로 승인한 경우에만 다음을 수행해:

1. 전체 Change Request 상태 재확인
2. 모든 Change Item이 PASSED인지 확인
3. Git remote와 배포 브랜치 확인
4. 변경 파일과 Git diff 검토
5. 변경 범위가 CHANGE_REQUEST.md와 일치하는지 확인
6. 예상하지 않은 파일 변경 확인
7. github_token.txt가 추적 또는 stage되지 않았는지 확인
8. env_settings.txt, .env 및 비밀정보 파일 확인
9. 필요한 경우 비밀정보 파일을 .gitignore에 추가
10. 코드, diff, 문서에 토큰 문자열이 없는지 검사
11. 필요한 파일만 stage
12. 전체 Change Request ID가 포함된 commit 메시지 작성
13. commit 생성
14. 현재 디렉토리의 github_token.txt에 있는 token="" 값을 안전하게 인증에 사용
15. GitHub Pages 대상 브랜치에 push
16. push 결과와 commit hash 확인
17. GitHub Pages 재배포 완료 확인
18. 배포 주소에서 HTTP 200 확인
19. 배포된 사이트에서 모든 변경사항 확인
20. 배포된 사이트에서 전체 회귀 테스트 재실행
21. 브라우저 콘솔 오류 확인
22. 모든 Change Item 상태를 DEPLOYED로 업데이트
23. MEMORY.md에 새 정상 배포 commit과 URL 기록
24. 최종 상태를 DEPLOYED로 설정

토큰 규칙:

- 토큰 값을 출력하지 마
- 사용자에게 토큰을 보여주지 마
- 코드에 토큰을 저장하지 마
- MEMORY.md 또는 CHANGE_REQUEST.md에 기록하지 마
- commit에 포함하지 마
- 영구 remote URL에 포함하지 마
- 로그나 명령 출력에 노출하지 마
- 인증용 임시 파일이나 환경 변수를 만들었다면 즉시 제거해

권한, 인증, GitHub Pages 설정 또는 저장소 문제가 발생하면
임의로 우회하지 말고 BLOCKED 또는 HITL_REQUIRED로 종료해.

## 19. 최종 보고

재배포 승인 전에는 다음을 보고해:

- 전체 Change Request ID
- Change Item별 상태
- 반영한 사용자 요청
- 변경 파일
- 참고한 문서와 자료
- 수정 전 재현 결과
- 수정 후 테스트 결과
- 웹사이트 회귀 테스트
- 게임 회귀 테스트
- 반응형 검증 결과
- 콘텐츠 검증 결과
- Retry 횟수
- Claude Code CLI 사용 여부
- 실제 사용 모델
- BLOCKED 항목
- HITL_REQUIRED 항목
- 현재 상태
- 배포 대상 저장소
- 예상 배포 주소

사용자가 재배포를 승인한 후에는 추가로 다음을 보고해:

- 새 commit hash
- push 결과
- GitHub Pages 주소
- HTTP 응답
- 배포된 사이트에서 확인한 Change Item별 결과
- 배포 후 전체 회귀 테스트
- 새 마지막 정상 배포 commit
- 최종 상태: DEPLOYED, BLOCKED 또는 HITL_REQUIRED
