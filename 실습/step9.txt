현재 프로젝트의 CHANGE_REQUEST.md, AORR.md, MEMORY.md, 현재 코드와 마지막 정상 배포 commit을 읽고 사용자 수정 요청에 대한 새로운 Self-Correcting TDD 재루프를 실행해줘.

이번 루프의 목표는 Step 8에서 확정한 사용자 수정사항만 안전하게 반영하고, 기존 프로페셔널 웹사이트와 지렁이 게임의 정상 기능을 유지한 상태로 검증하는 것이야.

모든 변경 테스트와 회귀 테스트가 통과하면 바로 배포하지 말고 사용자에게 재배포 승인을 요청해.
사용자가 명시적으로 승인한 경우에만 GitHub Pages에 재배포해.

기존 완료 기록, 실행 기록, 마지막 정상 배포 정보는 삭제하거나 덮어쓰지 마.

## 1. 변경 요청 확인

먼저 다음을 확인해:

1. CHANGE_REQUEST.md의 Change Request ID
2. 사용자 수정 요청 원문
3. 현재 동작
4. 기대 동작
5. 객관적인 완료 기준
6. 수정 허용 범위
7. 변경 금지 범위
8. 마지막 정상 배포 commit
9. 마지막 정상 배포 URL
10. 기존 테스트와 회귀 테스트
11. 현재 Retry 횟수
12. Git remote와 현재 브랜치

CHANGE_REQUEST.md의 요구사항이 불명확하거나 서로 충돌하면 코드를 수정하지 말고
상태를 HITL_REQUIRED로 설정한 뒤 [사람 확인 필요] 항목을 보고해.

## 2. 수정 전 상태 보존

수정하기 전에 다음을 기록해:

- 현재 commit hash
- 현재 Git 상태
- 마지막 정상 배포 commit
- 마지막 정상 배포 URL
- 수정 대상 파일
- 현재 동작의 재현 결과
- 기존 테스트 결과
- 변경 전 스크린샷 또는 화면 상태를 확인할 수 있다면 해당 결과

문제가 발생할 경우 마지막 정상 배포 commit으로 복구할 수 있도록
rollback 기준을 MEMORY.md에 기록해.

단, 사용자의 승인 없이 실제 rollback, reset, force push를 수행하지 마.

## 3. Self-Correcting TDD 재루프

변경 작업은 다음 순서로 수행해:

1. 수정 전 문제 또는 요청 사항 재현
2. 재현 결과를 MEMORY.md에 기록
3. 실패 테스트 또는 객관적인 검증 기준 확정
4. 실패 원인 분류
5. 가장 작은 수정 하나 수행
6. 동일한 테스트 또는 재현 절차 재실행
7. 사용자 수정 요청의 완료 기준 검증
8. 기존 프로페셔널 웹사이트 회귀 테스트
9. 기존 지렁이 게임 회귀 테스트
10. 모바일, 태블릿, 데스크톱 검증
11. GitHub Pages 호환성 검증
12. 결과를 MEMORY.md의 새로운 실행 기록으로 추가
13. 통과하지 못하면 Retry 정책 확인 후 다음 루프 수행

각 Retry에서는 하나의 실패 원인만 수정하고,
관련된 최소 파일만 변경해.

## 4. 실패 원인 분류

실패 원인은 다음 중 하나로 분류해:

- HTML_STRUCTURE
- CSS_RESPONSIVE
- JAVASCRIPT
- GAME_STATE
- GAME_LOGIC
- GAME_CONTROL
- CONTENT
- ACCESSIBILITY
- TEST
- ENVIRONMENT
- GITHUB_PERMISSION
- DEPLOYMENT
- UNKNOWN

각 실패에 대해 다음 정보를 기록해:

- 실행한 명령 또는 재현 절차
- exit code
- 핵심 오류 메시지
- 관련 파일과 라인
- 실패한 완료 기준
- 오류 fingerprint
- 수정 가설
- Retry 횟수

## 5. Retry 정책

다음 정책을 반드시 지켜:

- 하나의 오류당 최대 3회 Retry
- 동일 오류 fingerprint가 2회 반복되면 즉시 중지
- 한 번의 Retry에서는 하나의 원인만 수정
- CHANGE_REQUEST.md에 정의된 범위 밖의 파일은 수정하지 않음
- 범위 밖의 수정이 불가피하면 먼저 HITL_REQUIRED로 중지
- 테스트 삭제 또는 완료 기준 완화 금지
- 기존 기능을 제거해서 문제를 해결하지 않음
- 불필요한 전체 사이트 재작성 금지
- 기존 개인 콘텐츠 임의 삭제 금지
- 권한 또는 환경 문제를 코드 변경으로 우회하지 않음

Retry 한계에 도달하면 상태를 HITL_REQUIRED로 설정하고 중지해.

## 6. 필수 변경 검증

다음을 모두 검증해:

### 변경 요청 검증

- 사용자 수정 요청 원문이 모두 반영되었는지
- CHANGE_REQUEST.md의 완료 기준을 충족하는지
- 수정 범위가 승인된 범위를 벗어나지 않았는지
- 기대 동작과 실제 동작이 일치하는지

### 기존 웹사이트 회귀 테스트

- Home 또는 Hero 영역
- About 영역
- Projects, Research, Experience 등 기존 콘텐츠
- Contact 영역
- 상단 내비게이션
- 모바일 내비게이션
- 기존 외부 링크
- 기존 프로필 콘텐츠
- 깨진 내부 링크 여부
- 브라우저 콘솔 오류 여부

### 반응형 회귀 테스트

가능하면 다음 viewport에서 검증해:

- 모바일: 약 375px
- 태블릿: 약 768px
- 데스크톱: 약 1440px

다음을 확인해:

- 가로 스크롤 없음
- 텍스트와 UI 잘림 없음
- 내비게이션 정상 동작
- 버튼과 링크의 터치 영역 유지
- 게임 영역이 viewport를 벗어나지 않음

### 지렁이 게임 회귀 테스트

- Games 탭 정상 동작
- 게임 시작
- 일시정지 및 재개
- 다시 시작
- 자동 이동
- 음식 생성
- 음식 섭취와 성장
- 점수 증가
- 최고 점수
- 벽 충돌
- 자기 몸 충돌
- 게임 오버
- 방향키 조작
- WASD 조작
- 모바일 방향 버튼
- 구현된 경우 Swipe 조작
- 반대 방향 즉시 전환 방지
- 중복 게임 타이머 방지
- Games 탭을 반복해서 열어도 중복 실행되지 않음

변경 요청이 게임과 관계없더라도 기존 게임 기능의 회귀 테스트를 수행해.

### GitHub Pages 호환성

- 루트 index.html 존재
- HTML, CSS, JavaScript 정상 연결
- 상대 경로 사용
- 로컬 절대 경로 없음
- 백엔드 또는 데이터베이스 의존성 없음
- GitHub Pages에서 지원하지 않는 서버 기능 없음
- github_token.txt가 웹사이트 코드에 연결되지 않음
- 비밀정보 파일이 Git 추적 대상이 아님

브라우저 자동화나 실제 모바일 검증이 불가능한 항목은
통과한 것으로 추측하지 말고 [사람 확인 필요]로 표시해.

## 8. 로컬 변경 완료 조건

다음 조건을 모두 만족해야 로컬 수정 완료로 판단해:

- 사용자 수정 요청 완료 기준 충족
- 수정 전 문제 또는 요구사항 재현 결과 기록
- 수정 후 동일한 검증 통과
- 기존 프로페셔널 웹사이트 회귀 테스트 통과
- 기존 지렁이 게임 회귀 테스트 통과
- 모바일, 태블릿, 데스크톱 검증 완료
- 브라우저 콘솔에 치명적인 오류 없음
- 깨진 링크 없음
- GitHub Pages 호환성 검증 완료
- CHANGE_REQUEST.md와 MEMORY.md 업데이트 완료

하나라도 충족되지 않으면
PASSED, DEPLOY_READY 또는 DEPLOYED로 표시하지 마.

모든 로컬 변경과 검증이 이상 없이 완료되면:

1. CHANGE_REQUEST.md 상태를 DEPLOY_APPROVAL_REQUIRED로 업데이트
2. MEMORY.md 현재 상태를 DEPLOY_APPROVAL_REQUIRED로 업데이트
3. 아직 commit, push 또는 재배포하지 않음
4. 사용자에게 재배포 승인을 요청하고 중지

## 9. 사용자 재배포 승인 요청

Git remote를 확인해 실제 GitHub 사용자명과 저장소를 구해.

사용자에게 다음 형식으로 질문해:

“Change Request [Change Request ID]의 수정사항 반영과 로컬 검증이 완료되었습니다.

반영한 요청:
[수정 요청 요약]

변경 파일:
[변경 파일 목록]

대상 저장소:
https://github.com/[GitHub 사용자명]/[GitHub 사용자명].github.io

예상 배포 주소:
https://[GitHub 사용자명].github.io

현재 수정 버전을 GitHub Pages에 재배포할까요?”

현재 저장소가
https://github.com/daniel-eai/daniel-eai.github.io.git 인 경우에는 다음 주소를 사용해:

- 대상 저장소: https://github.com/daniel-eai/daniel-eai.github.io
- 배포 주소: https://daniel-eai.github.io

사용자가 명시적으로 승인하기 전에는 다음을 하지 마:

- commit
- push
- GitHub Pages 재배포
- GitHub Pages 설정 변경
- Git remote 변경

## 10. 승인 후 재배포

사용자가 명시적으로 재배포를 승인한 경우에만 다음 절차를 수행해:

1. Change Request ID와 현재 상태 재확인
2. Git remote에서 GitHub 사용자명과 저장소 재확인
3. 저장소가 [GitHub 사용자명]/[GitHub 사용자명].github.io 형식인지 확인
4. 변경 파일과 Git diff 검토
5. 변경 범위가 CHANGE_REQUEST.md와 일치하는지 확인
6. 예상하지 않은 파일이 변경되지 않았는지 확인
7. github_token.txt가 Git 추적 또는 stage 대상이 아닌지 확인
8. env_settings.txt, .env 및 기타 비밀정보 파일이 포함되지 않았는지 확인
9. 필요한 경우 비밀정보 파일을 .gitignore에 추가
10. 코드, Git diff, MEMORY.md에 토큰 문자열이 포함되지 않았는지 검사
11. 배포에 필요한 파일만 stage
12. Change Request ID가 포함된 commit 메시지 작성
13. commit 생성
14. 현재 디렉토리의 github_token.txt에 있는 token="" 값을 GitHub 인증에 사용
15. GitHub Pages 대상 브랜치에 push
16. push 결과와 새 commit hash 확인
17. GitHub Pages 재배포 완료 확인
18. https://[GitHub 사용자명].github.io 에서 HTTP 200 확인
19. 배포된 사이트에서 사용자 수정사항 확인
20. 배포된 사이트에서 전체 회귀 테스트 재실행
21. 배포된 사이트에서 브라우저 콘솔 오류 확인
22. CHANGE_REQUEST.md 상태를 DEPLOYED로 업데이트
23. MEMORY.md에 새 정상 배포 commit과 URL 기록
24. 최종 상태를 DEPLOYED로 설정

토큰은 다음 규칙에 따라 사용해:

- 토큰 값을 출력하지 마
- 토큰을 사용자에게 보여주지 마
- 토큰을 코드에 저장하지 마
- 토큰을 MEMORY.md에 기록하지 마
- 토큰을 Git commit에 포함하지 마
- 토큰을 영구 Git remote URL에 포함하지 마
- 토큰을 명령 출력이나 로그에 노출하지 마
- 인증용 임시 파일이나 환경 변수를 만들었다면 배포 후 즉시 제거해

저장소 형식이 다르거나,
GitHub Pages 설정 변경이 필요하거나,
인증 또는 권한 문제가 발생하면 임의로 우회하지 마.

상태를 HITL_REQUIRED 또는 BLOCKED로 설정하고
[사람 확인 필요]로 보고해.
