Step 7에서 배포된 GitHub Pages 웹사이트를 사용자가 검수한 뒤 수정 요청을 전달했어.

아래 사용자 요청을 특정 예시나 기능에 한정하지 말고 분석해.
버그 수정, UI/UX 개선, 반응형 문제, 성능 개선, 콘텐츠 변경, 문서 기반 콘텐츠 작성,
페이지 구조 변경, 게임 기능 추가, 접근성 개선, 배포 문제 등 어떤 형태의 요청도 처리할 수 있어야 해.

사용자 수정 요청:
"""
[배포된 웹사이트를 보고 수정하고 싶은 내용을 자유롭게 입력]
"""

추가 자료:
"""
[문제가 발생한 기기, 브라우저, 재현 방법, 참고 디자인, 추가 요구사항 등을 입력]
"""

사용자가 CV, 이력서, PDF, 이미지, 문서 또는 기타 자료를 언급했다면
현재 프로젝트와 현재 디렉토리에서 해당 파일을 찾아 읽고 변경 계획에 반영해.

파일이 없거나 어떤 파일을 의미하는지 불명확하면 추측하지 말고
[사람 확인 필요]로 표시해.

이번 단계의 목표는 사용자 수정 요청을 바로 구현하는 것이 아니라,
실행 가능한 여러 개의 Loop Engineering 변경 업무로 분해하는 것이야.

## 1. 현재 상태 확인

다음 자료를 먼저 확인해:

- 배포된 GitHub Pages 웹사이트
- 현재 소스 코드와 파일 구조
- Git remote와 현재 브랜치
- AORR.md
- MEMORY.md
- 마지막 정상 배포 commit
- 마지막 정상 배포 URL
- 기존 테스트와 실행 기록
- 사용자가 제공하거나 언급한 참고 파일
- 현재 Git 상태

기존 정상 배포 상태를 변경 전 기준선으로 설정해.

## 2. 사용자 요청 원문 보존

사용자 수정 요청을 원문 그대로 보존해.

다음을 하지 마:

- 사용자 표현 임의 생략
- 여러 요청을 하나로 뭉개기
- 요구사항의 의미를 임의로 바꾸기
- 모호한 내용을 확정된 요구사항처럼 해석하기

사용자 요청에 여러 수정사항이 포함되어 있다면
각 요청을 독립적으로 추적할 수 있도록 원자적인 Change Item으로 분리해.

예시의 게임이나 웹사이트 항목을 하드코딩하지 말고,
실제 사용자가 입력한 내용에 따라 항목을 동적으로 생성해.

## 3. 변경 요청 분해

각 수정사항에 고유한 Change Item ID를 부여해.

형식 예시:

- CR-001
- CR-002
- CR-003

각 Change Item에 다음 내용을 작성해:

- Change Item ID
- 사용자 요청 원문
- 요청 요약
- 요청 분류
- 현재 동작
- 기대 동작
- 재현 방법
- 근거 자료
- 수정 대상 기능
- 예상 수정 파일
- 변경 허용 범위
- 변경 금지 범위
- 선행 작업
- 후속 작업
- 다른 Change Item과의 의존성
- 완료 기준
- 검증 방법
- 회귀 테스트
- 위험도
- 배포 필요 여부
- 사람 확인 필요 항목

## 4. 요청 분류

각 Change Item을 다음 중 하나 이상으로 분류해:

- BUG
- UI_UX
- RESPONSIVE
- PERFORMANCE
- CONTENT
- DOCUMENT_BASED_CONTENT
- INFORMATION_ARCHITECTURE
- MULTI_PAGE_STRUCTURE
- NAVIGATION
- GAME_LOGIC
- GAME_CONTROL
- GAME_ENTITY
- GAME_EFFECT
- GAME_STATE
- ACCESSIBILITY
- NEW_FEATURE
- SPEC_CHANGE
- REFACTOR
- DEPLOYMENT
- SECURITY
- UNKNOWN

하나의 요청이 여러 분류에 해당하면 복수 분류를 허용해.

## 5. 참고 자료 기반 콘텐츠 처리

사용자가 CV, 이력서, PDF 또는 문서를 바탕으로 웹사이트 내용을 채우라고 요청한 경우:

1. 해당 자료의 존재 여부를 확인
2. 문서에서 확인 가능한 사실만 추출
3. 이름, 직함, 학력, 경력, 연구, 프로젝트, 논문, 기술, 수상, 연락처 등을 구분
4. 각 정보가 들어갈 페이지 또는 섹션을 결정
5. 문서에 없는 사실은 생성하지 않음
6. 문서에서 확실하지 않은 표현은 [사람 확인 필요]로 표시
7. 개인정보 또는 공개 여부가 불확실한 내용은 HITL 조건으로 분류

사용자가 웹사이트 분위기나 디자인을 자료에서 추론해 달라고 요청한 경우에는:

- 자료에서 드러나는 전문 분야
- 경력의 분위기
- 주요 연구 및 프로젝트 성격
- 사용자의 전문적 이미지
- 강조해야 할 핵심 주제

를 근거로 디자인 방향을 제안해.

단, 디자인 방향과 문서에 명시된 사실을 구분해서 기록해.

## 6. 구조 변경 처리

사용자가 단일 페이지를 멀티페이지로 변경하거나
내비게이션, 디렉토리 구조, 콘텐츠 구조를 바꾸라고 요청한 경우에는 다음을 포함해:

- 변경 전 정보 구조
- 변경 후 정보 구조
- 생성할 페이지
- 이동할 콘텐츠
- 공통 내비게이션
- 공통 CSS 및 JavaScript
- 페이지 간 링크
- GitHub Pages 상대 경로
- 기존 URL 및 앵커에 미치는 영향
- 깨진 링크 위험
- 모바일 내비게이션 영향
- 회귀 테스트 범위

구조 변경이 여러 파일을 동시에 수정해야만 완전한 상태가 된다면
이를 하나의 원자적인 구조 변경 루프로 묶을 수 있어.

## 7. 중복, 충돌 및 모호성 분석

사용자 요청 사이에서 다음을 검사해:

- 중복 요청
- 같은 원인을 가진 요청
- 서로 충돌하는 요청
- 기존 요구사항과 충돌하는 요청
- 구현 순서가 필요한 요청
- 다른 변경이 완료되어야 시작할 수 있는 요청
- 범위가 불명확한 요청
- 디자인 취향처럼 객관적인 완료 기준이 부족한 요청

중복 요청은 합칠 수 있지만 사용자 원문은 모두 보존해.

충돌하거나 모호한 요청은 임의로 결정하지 말고
HITL_REQUIRED 또는 [사람 확인 필요]로 표시해.

## 8. 완료 기준 작성

각 Change Item에는 객관적으로 검증 가능한 완료 기준을 작성해.

완료 기준은 요청 특성에 따라 다음을 포함할 수 있어:

- 재현하던 버그가 더 이상 발생하지 않음
- 입력 후 반응 시간이 정해진 기준을 충족함
- 특정 viewport에서 요소가 화면 안에 표시됨
- 키보드, 마우스, 터치 입력이 정상 동작함
- 새로운 기능의 시작, 진행, 종료 상태가 정상임
- 새로운 페이지가 생성되고 내비게이션으로 접근 가능함
- 참고 문서의 사실이 정확하게 반영됨
- 기존 콘텐츠가 누락되지 않음
- 모바일, 태블릿, 데스크톱에서 정상 동작함
- 접근성 기준을 충족함
- GitHub Pages 배포 후에도 동일하게 동작함

“더 예쁘게”, “더 빠르게”, “더 자연스럽게”처럼 모호한 표현은
검증할 수 있는 기준으로 구체화해.

구체화할 근거가 부족하면 [사람 확인 필요]로 표시해.

## 9. 테스트 계획

각 Change Item에 대해 다음 테스트를 정의해:

### 변경 전 재현 테스트

- 현재 문제를 재현하는 절차
- 기대되는 실패 결과
- 재현 환경
- 관찰할 로그 또는 화면

### 변경 후 테스트

- 수정 후 동일 절차
- 통과 기준
- 자동 검증 가능 여부
- 수동 검수가 필요한 항목

### 회귀 테스트

- 기존 프로페셔널 웹사이트
- 기존 페이지와 콘텐츠
- 내비게이션
- 모바일 반응형
- Games 탭
- 기존 게임 기능
- 키보드 조작
- 모바일 조작
- 점수 및 게임 상태
- 브라우저 콘솔
- 깨진 링크
- GitHub Pages 상대 경로

변경 요청과 관계없는 기능도 변경의 영향을 받을 가능성이 있다면
회귀 테스트에 포함해.

## 10. 위험도와 사람 확인 조건

각 Change Item의 위험도를 다음 중 하나로 설정해:

- LOW
- MEDIUM
- HIGH

다음 조건은 기본적으로 HITL 후보로 분류해:

- 요구사항이 모호함
- 기존 사용자 요청과 충돌
- 사실 확인이 필요한 개인정보
- 기존 콘텐츠 대규모 삭제
- URL 구조 대규모 변경
- 외부 서비스 또는 API 추가
- 보안 또는 인증 변경
- 새 라이브러리 또는 프레임워크 도입
- GitHub Pages 설정 변경
- 사용자의 디자인 선택이 필요한 경우
- 구현 범위가 최초 요청보다 크게 확장되는 경우

## 11. 변경 루프 설계

Change Item을 단순 목록으로만 만들지 말고
실행 가능한 AORR 변경 루프로 구성해.

각 루프에는 다음을 포함해:

- Loop ID
- 연결된 Change Item
- Target
- 입력 자료
- Act
- Observe
- Reason
- Verifier
- 완료 기준
- Retry 정책
- Stop 조건
- HITL 조건
- 예상 수정 파일
- 선행 Loop
- 다음 Loop

루프 상태는 다음 중 하나로 표현해:

- CHANGE_INTAKE
- CHANGE_PLANNED
- READY
- ACTING
- VERIFYING
- RETRYING
- PASSED
- BLOCKED
- HITL_REQUIRED
- DEPLOY_APPROVAL_REQUIRED
- DEPLOYED

## 12. 실행 순서 결정

모든 Change Item을 다음 원칙으로 정렬해:

1. 현재 문제 재현과 기준선 확보
2. 다른 작업을 방해하는 치명적인 버그
3. 입력 및 성능 문제
4. 구조와 페이지 아키텍처 변경
5. 콘텐츠 이동 및 문서 기반 콘텐츠 작성
6. 신규 기능
7. 시각 효과 및 디자인 개선
8. 접근성
9. 전체 회귀 테스트
10. GitHub Pages 호환성
11. 재배포 승인

실제 요청의 의존성에 따라 순서를 조정해.

서로 독립적인 작업은 별도 루프로 유지하고,
동시에 수정해야만 정상 동작하는 작업은 하나의 원자적 루프로 묶어.

## 13. 파일 업데이트

다음 파일을 업데이트해:

### CHANGE_REQUEST.md

다음을 포함해:

- 전체 Change Request ID
- 사용자 요청 원문 전체
- 참고 자료
- 기준선 commit 및 배포 URL
- Change Item 목록
- 요청 분류
- 현재 동작과 기대 동작
- 완료 기준
- 테스트 계획
- 회귀 테스트
- 의존성
- 실행 순서
- 위험도
- 예상 수정 파일
- 배포 필요 여부
- 사람 확인 필요 항목
- 현재 상태

### AORR.md

기존 내용을 삭제하지 말고 아래에 새로운 섹션을 추가해:

`Change Request Loop Plan`

각 변경 루프의 Target, Act, Observe, Reason, Repeat, Stop,
Verifier, HITL 조건과 상태 전이를 기록해.

### MEMORY.md

기존 기록을 삭제하거나 덮어쓰지 말고 다음을 추가해:

- 마지막 정상 배포 commit
- 마지막 정상 배포 URL
- 새로운 전체 Change Request ID
- Change Item 목록
- 사용자 요청 요약
- 참고 자료
- 현재 상태
- 새 완료 기준
- 루프 실행 순서
- 다음 Step 9에서 실행할 첫 번째 Loop ID
- Rollback 기준
- 사람 확인 필요 항목

## 14. 이 단계에서 금지할 작업

이번 단계에서는 다음을 수행하지 마:

- 웹사이트 코드 수정
- 게임 코드 수정
- 콘텐츠 파일 수정
- commit
- push
- GitHub Pages 재배포
- GitHub Pages 설정 변경
- 기존 테스트 결과 삭제
- 기존 실행 기록 삭제
- 사용자 요청에 없는 기능 임의 추가

## 15. 최종 보고

다음을 사용자에게 보고해:

- 전체 Change Request ID
- 생성된 Change Item 수
- Change Item별 분류
- 통합하거나 분리한 요청
- 참고한 자료
- 실행할 변경 루프 순서
- 각 루프의 완료 기준
- 위험도가 높은 항목
- 예상 수정 파일
- Step 9에서 즉시 실행 가능한 루프
- HITL_REQUIRED 항목
- [사람 확인 필요] 항목

요구사항이 충분히 명확하고 실행 가능한 경우
전체 상태를 CHANGE_PLANNED로 설정해.

핵심 요구사항이 불명확하거나 서로 충돌하는 경우
전체 상태를 HITL_REQUIRED로 설정하고 구현을 시작하지 마.
